一、核心展示区：三条真实消息主链切换

页面主视觉建议用同一块流程舞台承接 3 类模式切换。不是三份独立文档，而是一块统一消息舞台，让人直接看见“谁写入、谁补投影、谁负责展示”。

用户直发链

展示重点不是“发出去一条消息”，而是“一条 run 的双面展示”。这能直接解释为什么聊天区看起来像两条消息，底层却仍是同一 run。

写入
前端 composer 发 announce

最小字段通常只有 `sender_type=user / sender_name=我 / sessionId / message`，服务端补 `visible_in_channel_chat=true`。

  • 写入真源：run meta + 原始正文
  • 目标：先保证消息进会话时间线
  • 不默认补 `communication_view`
执行
CLI 执行继续写回同一 run

CLI 过程与最终输出继续写在 `partialPreview / lastPreview / detail`，没有再单独生成一条“普通回复消息”。

  • assistant 输出回填同一 run
  • 用户问答是同一 run 的双面状态
  • 最适合用“展开/折叠过程”来解释
展示
前端拆成用户气泡 + assistant 气泡

界面上看到的是“像一来一回”，但应该明确标注：这是单 run 渲染，不是天然双消息对象。

  • 适合加“同 run 双视图”说明条
  • 降低用户对结构的误读
  • 把 technical detail 收在第二层

Agent 协作链

这条链比用户消息结构化得多，应该在界面上被显式定义成“正式协作动作”，不是普通聊天消息。

派发
显式带协作字段入 announce

常见会带 `message_kind=collab_update`、`source_ref`、`target_ref`、`callback_to`、`interaction_mode`。

  • 天然比用户消息更接近 message contract
  • 适合界面上单独分层为“协作消息”
  • 可视化时要把回执预期画出来
处理
目标会话接单并执行

当前很多执行位会把结果写在被派发过去的入站 run 上，随后依赖系统 callback 自动回源。

  • 这就是“直接正式回包不稳定”的来源
  • 路由未必坏，习惯可能有问题
  • 适合在界面上标出 direct reply 与 callback 的差别
回源
源会话感知到系统回调而非直接回包

如果展示层不区分 direct reply 和 callback，用户就会误以为前端已经稳定直回。

  • 建议首屏直接给“回包方式”标签
  • 把 callback_auto 单独着色
  • 对缺 direct reply 的任务做弱提醒

系统 callback 链

这是当前最完整、最稳定的一支，所以方案页里要把它做成“结构化消息标准样板”，用于反衬普通 announce 的不足。

构造
运行时自动找回送目标

callback 会根据来源链和目标链，构造回执正文与回送目标，属于系统层主动动作。

  • 天然带 `trigger_type=callback_auto`
  • 发送者通常是 `system`
  • 更适合做正式回执的摘要节点
补投影
补齐 communication_view / receipt_summary

这条链当前会稳定补富结构字段，所以最适合作为“消息对象”的界面模板。

  • 可直接画“当前结论 / 是否通过 / 下一步动作”
  • 可挂 receipt items / pending actions
  • 不需要前端猜太多
回挂
回到源消息下方形成回执链

界面应该明确它是“系统生成的回执层”，而不是普通聊天继续往下堆。

  • 建议用“回执层”样式而非普通气泡
  • 帮助用户一眼识别完成态
  • 也方便审计消息闭环

二、侧边透镜：为什么现在会看起来模糊

写入真源统一，但投影真源不统一 `announce` 统一了写入入口，但没有统一成一套稳定的富消息对象。页面上要直接画出“写入层 / 运行时投影层 / 展示补偿层”。
普通 announce 轻，system callback 富 这是当前最关键的反差。方案页要用颜色和字段密度直接表现，不再让用户从长文里自己推。
前端承担了大量语义猜测 现在很多“看起来像完整消息”的卡片，其实是 UI 读路径补出来的，不是后端保证的稳定结构。
direct reply 与 callback 容易混淆 尤其是 Agent 协作链，目标会话写完入站 run 后靠系统自动回源，业务上容易被误判成“已经直接回包”。

三、字段差异对照区

这块建议直接用一张对照表表达“哪类消息天然带什么”。重点不是字段全列，而是让业务人员一眼看懂：哪些能力是稳定保证，哪些只是当前偶然可见。

消息类型 稳定拥有 当前缺口
用户直发run + 原始正文 + `visible_in_channel_chat` 基础入站、最小可见性、同 run 双面展示 通常不自动带 `communication_view / receipt_summary`
正式协作`message_kind + source_ref + callback_to` 来源链、目标链、回执预期更完整 处理完成后不一定有 direct reply run
系统 callback`trigger_type=callback_auto` `communication_view / receipt_summary` 最稳定 容易被误解成普通协作消息本来就这么完整

四、页面模块建议

模块 1：消息总览 Hero 一屏内给出统一入口、三条主链、当前最稳定消息族、当前模糊原因。让用户先建立全局认知。
模块 2：消息舞台 Flow Canvas 通过切换按钮展示三条链，强调每条链的写入、执行、展示三个阶段。
模块 3：结构差异表 把普通 announce 与 system callback 的字段覆盖差异做成对照矩阵，方便评审时快速拍板。
模块 4：当前风险与后续收敛 只列 3 到 5 条最重要的风险，例如“普通消息轻、前端补偿重、direct reply 易误判”，避免页面再次变成长文报告。

五、这页不是最终实现，而是给实现前讨论用的目标展示方式

如果后续要继续深化，这一页可以自然扩成“消息体系观察台”：左侧是消息类型切换，中间是流程舞台，右侧是字段真源和风险点。 但在当前阶段，建议先把它作为讨论页面，不直接混进正式 task 页。

delivery_state: 方案态 吸收基线：task-shell / request diagnostics / message flow map 用途：继续讨论消息体系，不直接替代正式页面
