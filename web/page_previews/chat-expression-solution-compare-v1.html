拍板摘要

一句话问题
首屏主语漂移，用户先看到技术收纳，不是业务推进对象
当前聊天区不是漏显示，而是原消息、下一动作和业务对象被摘要、回执和过程层抢走了首屏主位。
必须保留
  • 聊天区首先仍然是业务协作流，不回退成日志瀑布。
  • 反馈、过程、调试继续保留，只降到第二层。
  • 不动 live 接口，不新增运行时装配层。
本轮拍板点
  • 原消息主语是否恢复首屏直显。
  • 当前结论 / 当前状态 / 下一动作 是否成为固定首层。
  • 回执、过程、调试是否统一下沉为第二层入口。
问题定义
聊天区现在能展示很多信息，但用户读完首屏后，未必能立即判断“谁接下一棒”。
这页只解决“怎么拍板”问题：把现状为什么难读、方案态如何恢复业务主语、哪些技术层不会消失但会降层，一次说清楚。
目标态
原消息主语首屏直显 当前结论直显 当前状态直显 下一动作直显
本轮不做
不改 live 聊天区 不新增接口 不扩到 overview / 架构图 / Agent 工作台
现状与方案对照清单
先把功能、问题、计划方案和变更边界放在同一屏里，避免后续真实实现时偏题。
现状功能
当前已经有的能力
  • 会话预览与业务摘要主位。
  • 原消息“展开全文”与原文补拉。
  • 回执收纳卡与详情查看器。
  • 过程 / 调试抽屉与运行状态条。
当前问题
业务表达最伤的 4 件事
  • 原消息主语首屏不稳定。
  • 下一动作后移到摘要卡或回执明细里。
  • 反馈与原消息距离太近，技术语义抢主位。
  • 用户需要翻层后才能判断当前推进情况。
计划方案
只收敛到一个方向
  • 原消息或原协作对象恢复首屏直显。
  • 同层固定显示当前结论 / 状态 / 下一动作。
  • 回执、过程、调试统一降为第二层入口。
  • 保留摘要体系，但不再抢走首屏业务主语。
变更边界
本轮不会碰的东西
  • 不改 `task_tracking` 真源与运行时契约。
  • 不新增新的 viewer / drawer 体系。
  • 不重做聊天页主结构。
  • 不把技术状态条继续做成首屏主角。

实施保护栏
后续如果进入真实实现，这 3 类判断是保护栏，不允许因为修业务表达而把原有目标做丢。
必须保留
业务协作流仍是第一视角
  • 摘要体系继续存在，但退回“辅助判断”角色。
  • 原消息、原协作对象和下一动作必须能直接找到。
必须降层
反馈 / 过程 / 调试回到第二层
  • 技术明细不是删除，而是移到需要时再展开。
  • 首屏默认不再把“回执收纳 / 过程按钮”当主语。
暂不改动
运行时与读模型保持现状
  • 不新增接口、不新增运行时装配层。
  • 先按现有真源拍板，再拆真实前端修复批。
