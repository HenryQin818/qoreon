消息类型 Type

这套机制里，通道可见的消息主要分三类：用户输入、Agent 协作消息、系统自动回执。

user / manual_update agent / collab_update system / system_callback
交互模式 Mode

是否回执，不取决于“有没有说话”，而取决于消息被定义成哪种交互模式。

dialog_now task_with_receipt notify_only

其中 task_with_receipt 是强回执模式，最适合任务推进和验收前置。

路径字段 Path

看路径关系时，重点盯住这三个字段：源头在哪里、结果回到哪里、结果怎么被投回。

source_ref callback_to run_id

从用户到回执的最短闭环
你前面实际看到的那类消息，大体就是这条链路在跑。先发起，再执行，再自动回投，最后进入归档或验收层。

完整链路
把每一层的生成方式直接摊开
1
user / manual_update
用户先把业务意图说出来
这里是原始需求层。人不需要知道内部字段，只需要把问题说清楚。当前样例里，这一步就是你说“不要方案页 demo，要真实页面静态镜像预演页”。
输入层 人工发起
2
agent / announce_to_channel
源 Agent 把用户意图包装成正式协作消息
这一步会调用正式外发入口，把消息写成可追踪的 run，并带上 source_ref、callback_to 和 interaction_mode。它是可见消息，不是内部草稿。
可见聊天 任务派发
3
agent / task_with_receipt
目标 Agent 负责执行任务并给出结构化结果
这里是模型真正产出业务内容的地方。它会写结论、进展、阻塞、下一步等，最后形成一份结构化回执正文，供系统回投。
业务内容 结构化回执
4
system / system_callback
运行时自动生成回执壳并投回原会话
这一步不是 Agent 自己再发一条，而是 runtime 把已经完成的结果封装成系统回执。若遇到同 session 同 channel，则会被 self_suppressed，避免自回路消息。
自动投递 route_resolution
5
sink / archive
原会话和任务文件一起完成收口
消息回执解决的是“实时协作闭环”，任务文件和归档层解决的是“长期可追溯”。两者是上下游，不是同一个对象。
收口 证据沉淀
自动生成的是哪些 System
自动的是“壳”和“路由”，不是业务判断本身。
  • run 记录、状态、路由解析。
  • system_callback 回投卡片。
  • receipt_summary 和投递视图。
  • UI 里的系统分类与落盘记录。
Agent 生成的是哪些 Model
Agent 负责“想清楚”和“写清楚”。
  • 任务判断、执行过程、最终结论。
  • 结构化回执正文和 progress。
  • 内部子 Agent 的推理结果。
  • 必要时再发下一条正式消息。
用户怎么介入 Human
用户要改的不是“卡片文案”，而是“路由和模式”。
  • 决定要不要回执。
  • 决定回给谁，是否是 callback_to。
  • 决定任务是否是 task_with_receipt。
  • 决定是否需要对外可见消息。

核心结论
这套机制对 Agent 是可用的，对用户是可解释但不够直观的

你最需要看到的不是每一条聊天本身，而是“哪一层在生成内容、哪一层在自动投递、哪一层负责归档”。看懂这个之后，用户介入流程就会变成修改路径和模式，而不是面对一串难以拆开的消息。

阅读说明
先看路径，再看自动与手动边界。
这页怎么读

先看中间的链路图，再看左侧的消息类型与交互模式，最后看右侧的解释。这样你就能把“谁写内容”与“谁做投递”分开看。

最容易混淆的是 Agent 的最终结论 和 system_callback。前者是业务内容，后者是系统封装后的回投层。
为什么用户会觉得难介入

因为用户看到的是多层对象叠在一起：任务、消息、回执、运行时、归档。要真正介入，用户不是改一句话，而是改 interaction_mode、callback_to、source_ref 这些路径参数。

这就是你前面说的那个感觉：Agent 端能顺滑运转，用户端却不容易直接判断哪里该改、该从哪条路径介入。
自动回执的边界

运行时负责自动回投，但不会替代模型完成业务判断。也就是说，“自动”发生在投递层和路由层，不发生在业务理解层。

如果你之后要把这页继续推向可操作版本，下一步不是加更多消息，而是把“用户能改的点”单独做成开关：是否回执、回给谁、是否内部协作、是否归档。
和业务对齐的价值

这页的目标不是展示聊天记录，而是帮你判断：当前链路是否已经足够稳定，哪里是 Agent 的工作边界，哪里是系统自动工作边界，哪里是用户真正需要介入的边界。
