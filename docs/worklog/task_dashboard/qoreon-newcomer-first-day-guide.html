Qoreon 使用手册：
项目、通道、Agent 与跨项目协作

这不是一篇情绪化日志，也不是一堆内部术语堆出来的说明。它只解决一个问题：第一次看见 Qoreon 的人，能不能在一页里看懂这套系统是什么、里面有哪些角色、可以做哪些动作，以及一件事情是怎么被协同推进下去的。

一句话理解

Qoreon 是把项目、通道、Agent、对话、任务、消息、验收和跨项目桥接放进同一套工作现场里的协作系统。

System Structure

01. 先看整体，这套系统是怎么搭起来的

Qoreon 不是一个大聊天框。它是按“项目 -> 通道 -> Agent -> 对话 -> 任务 -> 协作链路”的层次组织起来的，层级不同，负责的问题也不同。

项目最大的工作容器

每个项目都有自己的任务空间、通道结构、会话真源、协作通讯录和运行上下文。项目先把边界划清楚，后面的任务和协作才不会混线。

通道项目内的职责槽位

通道负责长期分工，例如总控、架构治理、运行时、前端、测试、PMO。通道不是临时文件夹，而是稳定责任面。

Agent通道里的执行主体

Agent 可以接任务、接消息、持续在对话里推进工作，也可以联系其他 Agent。一个通道可以有主负责 Agent、备份 Agent 和专项 Agent。

对话真正开展工作的窗口

对话不只是聊天记录，而是操作面。你可以直接安排工作、发调查、要求输出方案、要求联系其他 Agent 并回执。

任务正式工作主线

需要明确推进、交付和验收的事情，要落到任务里。任务决定主线，消息推动过程，产出物留下证据，验收负责收口。

会话真源

真正工作的不是页面截图，而是 Agent 对应的会话与上下文。会话一旦绑定，就意味着这个角色开始持续积累自己的判断和工作历史。

消息链路

消息负责过程协作。一个 Agent 可以给另一个 Agent 发消息，带目标、带回执要求、带来源身份，而不是靠人手工转述。

证据链

产出物、run、回执、验收结论会留下来。这样一件事不是“说过了就算”，而是能回看、能接力、能核验。

Capabilities

02. 在 Qoreon 里，你到底可以做什么

如果只把它当聊天工具，你会低估它。如果只把它当任务看板，你也会误会它。它真正的能力在于：让“安排、执行、协同、回执、收口”发生在同一套现场里。

看项目和通道

你可以先从项目看全局，再进入单个通道看当前主线、活跃 Agent、最近推进和状态分布，知道现在这套系统正在忙什么。

多项目视角通道分工活跃状态

直接跟 Agent 对话

可以直接问情况，也可以直接安排动作：继续推进、先调查再回、生成任务、整理材料、汇总结论，不需要切出去再回填。

问进度安排行动持续上下文

让 Agent 自动去协作

不是所有事情都要你亲自转述。主责 Agent 可以联系其他 Agent，请他们补接口、核字段、回验收、给判断，再把结果带回主线。

消息派发回执回收链路继续

把工作落成任务

重要工作可以正式落成任务，写清楚目标、主责、状态、交付路径和验收口径。这样这件事就不再只是聊过，而是进入了可跟踪主线。

状态明确主责清楚可验收

追踪 run 与证据

协作过程不是黑箱。你可以看到 run、消息回执、产出路径和验收结论，知道一件事到底做到哪、卡在哪里、有没有真正被收口。

回执run证据可回放

跨项目形成机制

这套系统不只支持单项目内部协同，也支持项目与项目之间的结构化沟通。重点不是“互相发消息”，而是来源、目标和回执都不会混乱。

Workflow

03. 一件事情在这套系统里通常怎么被推进

Qoreon 的重点，不是把所有角色塞进一个对话框，而是让他们围着同一条主线各司其职，然后把结果继续往前推。

1

提出目标

事情通常从产品、用户、总控或某个板块需求开始。第一步不是写代码，而是先确认目标、边界和优先级。

2

定主线

总控或主责人决定这是不是正式主线，谁是主负责通道、谁是主负责 Agent，哪些支撑通道需要加入。

3

分动作

主责 Agent 开始推进，需要别的通道时，不是人肉转述，而是直接发结构化消息，让协作对象接下一棒。

4

收证据

过程中形成消息回执、run、产出物和判断结论。重要证据要能被别人接着看，而不是只停留在某个人脑子里。

5

做收口

最后由测试、验收、总控或主责人判断是否放行、是否归档、是否继续下一轮。没有收口，这件事就还没真正结束。

Recommended Organization

04. 我们推荐怎样组织工作，系统才会稳定

这套系统最怕两种极端：一种是所有事都回到总控，另一种是所有人都在游击推进。更稳的方式，是业务主责、平台支撑、中央门禁三层分开。

总控层

负责主线、优先级、是否放行和最后收口。它控制方向和节奏，但不应该长期下沉替代执行。

  • 控制任务层主线与批次
  • 处理跨板块冲突与取舍
  • 决定何时收口、何时升级

架构师 / 技术门禁

负责技术路线、契约边界、兼容与发布门禁。它保证工程秩序，不替代所有人持续写实现。

  • 控制技术边界与结构演进
  • 把高风险变更挡在门外
  • 给复杂实现提供路线判断

业务主责通道

承接长期板块推进。每个板块最好有主负责通道、主负责 Agent、备份 Agent 和支撑通道，避免知识反复丢失。

  • 对话/会话管理
  • 项目页/任务页
  • 自动化与桥接

日常管理与 PMO

负责节奏、催办、问题聚合、待验收清理，让系统不失速。它解决推进秩序，不替代业务和技术判断。

  • 催回执
  • 清积压
  • 盯状态一致性
Cross-Project Collaboration

05. 跨项目沟通不是混着聊，而是结构化桥接

Qoreon 不只适合单项目。它同样支持生产、开发、开源执行等多个项目之间的协作。但前提是：跨项目必须带结构化身份，不能把真源和上下文混写。

跨项目协作成立的前提，不是“能发到对方”，而是“回得回来，而且不会写错地方”。 推荐通过桥接通道或桥接角色，维持来源项目、目标项目、callback_to 与 run 证据一致。

跨项目协作的基本规则

  • 明确来源项目和目标项目
  • 明确来源通道、目标通道、主负责 Agent
  • 消息里带回执目标，保证处理完能回到原发起方
  • 桥接通道只负责链路，不替代目标项目实施
  • 禁止把 A 项目的真源直接写进 B 项目
Recommended Modes

06. 最后，推荐三种最好用的工作方式

如果你刚开始使用这套系统，不用一上来把所有能力都开满。先按这三种模式理解，就已经能覆盖大部分真实工作。

单项目主线模式

最常用。总控定主线，架构定边界，主责通道推进，支撑通道补能力，最后测试和验收收口。

日常功能推进项目内协同

多 Agent 联动模式

适合一件事同时牵动多个通道。主负责 Agent 持主线，其它 Agent 通过结构化消息补各自那一段，再回到主线汇总。

并行协作消息回执

跨项目桥接模式

适合生产、开发、开源执行等多个项目并行。通过桥接角色保证身份、上下文、回执和真源都不串线。

多项目联动桥接通道

最后一句话总结

如果你把 Qoreon 只理解成“会聊天的任务系统”，会低估它；如果你把它只理解成“能列任务的看板”，也会误会它。它真正的价值，是让项目、分工、Agent、对话、任务、证据、验收和跨项目协作都进入同一套可继续推进的工作机制。

当前页面口径基于项目现有组织方式、主责制模型、消息主驱动协作规则与跨项目桥接实践整理。
