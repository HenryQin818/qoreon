基础项目开发如何做分工：
Qoreon 通道分工方法

这篇文章不讲“通道越多越专业”，而是回答一个更有用的问题：一个基础项目开发，到底应该怎样分工，哪些通道是最小必需，哪些是按复杂度扩展，什么时候该建通道，什么时候其实只该建 Agent 或建任务。

核心结论

推荐采用 `2 + 4 + X` 分工模型：`总控 + 架构` 两个中央角色，`运行时 + 前端 + 数据 + 测试` 四个核心执行通道，再按项目复杂度扩 `PMO / 运维 / 原型 / 技能 / 巡检 / Git / 桥接 / 用户镜像` 等横向支撑。

适用阶段

新项目起步、分工失序重整、开源版最小通道集设计、已有多 Agent 但责任面混乱的阶段。

这篇的价值

不是罗列 Qoreon 当前 22 个通道，而是从方法论上讲清楚：什么是长期职责，什么是一次性事项，什么才配叫一个独立通道。

Why

01. 为什么项目分工会越来越乱

项目一复杂，组织方式如果还停留在“谁空谁上”，就会开始失控。最典型的问题其实就三类。

1

知识沉淀不下来

没有固定责任面时，同一块业务每轮都要重新理解。人一换，背景就断，推进会越来越慢。

2

责任边界会变糊

出了问题先找谁、谁来补证据、谁来收口都说不清。最后总控和架构被迫接住所有烂尾。

3

通道会越建越碎

如果没有判断门槛，一个临时问题就建一个通道，最后组织成本会高于问题本身。

Model

02. 推荐采用：2 + 4 + X 的分工模型

这不是“唯一正确答案”，但对于大多数基础软件项目，它已经足够稳，也足够克制。

2 个中央角色

项目必须先有中央层，负责控制方向和边界，而不是长期替代执行。

总控 / 产品主线 架构 / 技术门禁
  • 控目标与优先级
  • 控边界与门禁
  • 控冲突裁决与收口

4 个核心执行通道

这是一个基础软件项目最小可运行的交付面，已经覆盖了大部分真实工作。

运行时 / 后端 前端 / 交互 数据 / 契约 测试 / 验收
  • 后端负责能力与链路
  • 前端负责可见结果
  • 数据负责字段与兼容
  • 测试负责独立终验与放行

X 个按需扩展通道

当项目复杂度上升时，再把横向支撑能力从主线执行里剥出来。

PMO 结构治理 技能治理 原型承接 项目运维 Git / 开源 自动化巡检 跨项目桥接 用户镜像
Duties

03. 每一类通道分别负责什么，不负责什么

这部分一定要写清楚。因为很多协作问题，不是能力不够，而是职责边界没冻结。

总控 / 产品主线

决定现在推进什么，什么优先，什么时候收口。

负责
  • 主线和批次
  • 优先级与放行
  • 跨板块冲突裁决
不负责
  • 长期替代执行通道实现
  • 把所有问题都留给自己

架构 / 技术门禁

控制技术路线、结构边界、契约风险和高风险变更门禁。

负责
  • 技术路线判断
  • 结构边界冻结
  • 发布与回滚门禁
不负责
  • 持续替每个模块写代码
  • 承接全部日常推进

运行时 / 后端

负责 server、调度、API、并发、安全和执行环境。

输入
  • 需求与接口约束
  • 架构门禁
输出
  • 后端实现
  • 接口能力
  • 运行证据

前端 / 交互

负责页面结构、信息层级、导航组织和用户看得到的结果。

输入
  • 业务目标
  • 原型与接口
输出
  • 页面与交互
  • 可见验证结果

数据 / 契约

负责字段口径、数据一致性、兼容策略和回滚约束。

负责
  • 字段与白名单
  • 契约校验
  • 兼容治理
不负责
  • 替前后端推进全部实现
  • 替总控判断业务通过

测试 / 验收

负责独立终验、最小放行检查，以及“是否通过”的清晰结论。

负责
  • 功能 / 回归验证
  • 阻塞识别
  • 放行意见
不负责
  • 替执行方补实现
  • 把自测当终验

Decision Gate

04. 什么时候应该新建通道，什么时候其实只该建 Agent 或建任务

这是最容易做错的一层。组织一旦膨胀得比问题还快，系统就会越来越难管理。

该建通道

满足任意两条，才建议新建通道：

  • 工作会长期存在
  • 有稳定输入和输出
  • 需要独立知识沉淀
  • 会反复服务多个主线
  • 继续塞进现有通道会造成混责

不该建通道

下面这些通常只需要建任务或建 Agent：

  • 只是临时任务
  • 只是某通道里的专项动作
  • 只是某个 Agent 忙不过来
  • 只是补一轮讨论、联调或验收

一条判断公式

长期职责，建通道。
同一职责下的不同执行者，建 Agent。
这轮要做的一件事，建任务。
Qoreon Mapping

05. 用这套模型回看 Qoreon 当前体系，会发现什么

Qoreon 现在已经不是一个只有前后端的小项目。它的中央层、执行层、治理层和桥接层都已经比较明显，问题不在“有没有通道”，而在“哪些通道该承担长期责任、哪些只是支撑位”。

中央层

控制主线与技术边界
主体-总控（合并与验收） 子级01-Build引擎（架构师）

这两个位置已经承担了典型的“总控 + 架构”角色。后续应继续强化门禁职责，而不是下沉成长期执行替代者。

核心执行层

最像基础项目开发四件套
子级02-CCB运行时 子级04-前端体验 子级06-数据治理与契约 子级08-测试与验收

这四个通道已经构成最小执行面。对外讲 Qoreon 的基础研发组织，优先就讲这四个，不要一上来铺 20 多个通道。

平台增强层

不是基础交付，但对平台很关键
子级03-多CLI适配器 子级05-任务巡检与留痕 子级07-可视化看板

它们承担的是 Qoreon 的平台化特征，而不是传统项目都会有的基础研发职责。

支撑治理层

让系统可复制、可维护、可扩展
辅助01-项目结构治理 辅助02-团队协作Skills治理 辅助04-原型设计 辅助05-PMO 辅助06-项目运维 辅助07-Git仓库管理

这组通道说明 Qoreon 已经从“做功能”进入“做系统”的阶段，所以治理支撑已经不是可有可无，而是必要层。

特殊协作层

不是传统研发必备，但对当前系统很有价值
辅助03-用户镜像 辅助08-开源执行桥接 主体04-开源筹备联控（65主操作面）

一个偏业务判断和规划支撑，一个偏跨项目桥接和开源执行联控。这是 Qoreon 特有的组织能力。

当前最值得收敛的点

不是继续加通道，而是稳住主责和门禁
  • 先固定板块主责
  • 让支撑通道回到支撑位
  • 让总控和架构更多做门禁
  • 把跨项目桥接继续独立化
Flow

06. 一条需求应该怎样穿过这些通道

真正稳定的协作，不是所有通道都参与，而是让“主线、支撑、门禁、收口”各自站对位置。

1

提出目标

用户、产品或业务提出目标，先判断是不是正式主线。

2

总控定主线

决定优先级、批次和主负责通道，不让问题一上来就全局扩散。

3

架构定边界

冻结高风险约束、技术路线和契约边界，减少后面返工。

4

执行通道推进

后端、前端、数据等主线通道开始推进，形成实际交付。

5

支撑通道介入

当需要结构、技能、原型、运维、Git 等能力时，再接入横向支撑。

6

测试与验收

由独立验收通道判断是否通过，明确唯一阻塞和放行意见。

7

跨项目桥接

如果涉及生产、开发、开源执行等多项目，再由桥接通道建立正确链路。

Anti-Patterns

07. 这套分工里最常见的坏模式

如果你已经感觉系统很忙、通道很多、结果却还是不顺，通常不是能力不足，而是落进了这些模式。

总控长期包办执行

短期看很快，长期一定堵。总控一旦兼任全部主责，系统会把所有复杂度都挤到一个点上。

架构师变成全栈打杂

架构如果长期承担所有模块实施，就没时间做真正重要的边界和风险控制。

支撑通道替代业务主责

结构、PMO、运维、技能治理都是支撑位，不该反客为主替业务通道长期推进业务本身。

一出问题就新建通道

没有长期职责判断，就会把通道当临时垃圾桶，组织成本会迅速超过问题本身。

跨项目直接混写真源

项目之间一旦不通过桥接层而直接混写，会让来源、回执、上下文全部失真。

过程只靠反馈文件

过程协作不走消息主驱动，就会出现任务已完成、反馈仍待验收的双状态并存问题。

Takeaways

08. 最后，给基础项目开发的三条直接建议

如果你现在正在设计项目分工，不用一上来建满体系，先把最关键的骨架立住。

先立住中央层

先有 `总控 + 架构`，再谈执行通道。没有中央层，后面所有通道都会变成谁在忙谁接锅。

先跑通基础四件套

先把 `后端 + 前端 + 数据 + 测试` 跑顺，再按需要增加治理和支撑，不要反过来。

扩组织前先问一句

这到底是长期职责，还是这轮事项？回答不清楚，就先别建新通道。

一句话总结

真正有效的分工，不是通道越多越高级，而是主线有人负责、支撑有人承接、门禁有人把关、收口有人拍板。Qoreon 当前的价值，不只是已经有很多通道，而是它已经长出了把这些层次组织起来的基础骨架。
