# Qoreon 使用手册：项目、通道、Agent 与跨项目协作

适用对象：
- 第一次接触 Qoreon 的产品、研发、运营、项目协作者
- 想快速看懂这套系统到底能做什么、应该怎么组织工作的人

这不是入职感想，也不是一份只给内部人看的术语表。
它的目标很直接：让第一次接触这套系统的人，能在一页里看懂 Qoreon 的基本结构、常用动作、推荐协作方式，以及跨项目协作是怎么成立的。

## 1. 先看整体：Qoreon 到底是什么

Qoreon 不是单纯的聊天工具，也不是只有任务列表的看板系统。
它更像一个“可协作的项目操作面”，把下面几件事放进了同一套工作现场里：

1. 项目视角：你可以同时管理多个项目，而不是只盯一条任务流。
2. 通道视角：每个项目可以拆成多个通道，按职责分工推进。
3. Agent 视角：每个通道可以绑定主负责 Agent，持续接任务、回消息、沉淀上下文。
4. 对话视角：Agent 不是静态配置，而是可以直接对话、接收安排、持续执行的工作单元。
5. 任务视角：重要工作会落成任务，形成明确的状态、交付路径和验收口径。
6. 协作视角：Agent 之间可以互发消息、带回执协作，不需要所有事情都回到人手工传话。
7. 证据视角：产出物、run、消息回执、验收结论可以被保留下来，方便回看和继续接力。

一句话理解：
**Qoreon 是把“项目、分工、Agent、消息、任务、验收”组织到同一个工作界面中的协作系统。**

## 2. 这套系统里的核心对象是什么

### 2.1 项目
项目是最大的工作容器。
每个项目都有自己的：

1. 任务空间
2. 通道结构
3. 会话真源
4. 协作通讯录（CCR）
5. 运行上下文

项目的意义，是把一整套工作边界圈清楚。不同项目可以独立运行，也可以被桥接起来协同。

### 2.2 通道
通道是项目内部的分工单位。
你可以把它理解成“一个长期职责槽位”，例如：

1. 总控
2. 架构/结构治理
3. 前端实现
4. 运行时与 API
5. 测试与验收
6. PMO / 日常管理

通道的意义，不是多一个文件夹，而是让工作有稳定责任面。

### 2.3 Agent
Agent 是通道里的执行主体。
一个通道里可以有主负责 Agent，也可以有备份 Agent 或专项 Agent。

Agent 的作用是：

1. 接收任务
2. 在对话中持续推进
3. 给出结论和回执
4. 与其他 Agent 发消息协作
5. 保留本通道上下文

### 2.4 对话
对话是 Agent 的工作窗口。
Qoreon 里的对话不只是“聊天记录”，而是可以真正开展工作的操作面。

在对话里你可以：

1. 直接询问 Agent 当前情况
2. 安排它继续推进某条工作
3. 让它生成任务、整理结论、输出方案
4. 让它联系其他 Agent 去协同

### 2.5 任务
任务是正式的工作主线。
凡是需要明确推进、交付和验收的事情，都应该落到任务上。

一条任务至少应说清楚：

1. 目标是什么
2. 主负责通道和主负责 Agent 是谁
3. 交付物放在哪
4. 验收口径是什么
5. 当前状态在哪一档

### 2.6 消息与产出物
消息负责过程协作，产出物负责留下证据。

当前推荐口径是：

1. 任务为主线
2. 消息为过程
3. 产出物为证据
4. 反馈文件只在正式验收、归档、收口时使用

这意味着：
- 平时推进优先靠消息继续流动
- 需要保留结果时，再落产出物和验收结论

## 3. 用 Qoreon 你到底可以干哪些事

### 3.1 看项目
你可以从首页看多项目概况，知道：

1. 现在有哪些项目
2. 每个项目大概有哪些通道
3. 哪些项目当前活跃
4. 哪些项目正在推进、验收或开源发布

### 3.2 进项目看任务和通道
进入单个项目后，你可以继续往下看：

1. 项目下有哪些通道
2. 每个通道现在有没有活动任务
3. 当前任务状态分布如何
4. 哪些 Agent 在活跃工作

### 3.3 跟 Agent 对话
你可以直接在对话里跟 Agent 开展工作，不需要先切到其他系统再回来同步。

常见动作包括：

1. 问进度
2. 问判断
3. 让它继续推进当前任务
4. 让它先调查再回结论
5. 让它总结最近工作并沉淀成材料

### 3.4 给对话安排工作
Qoreon 不只是“问问题”，它支持把工作直接安排给对话中的 Agent。

例如：

1. 让某个 Agent 调查问题
2. 让它生成一版任务拆解
3. 让它去修一个具体板块
4. 让它根据已有产出做验收评估

### 3.5 让 Agent 去联系其他 Agent
这是这套系统很关键的一层能力。

不是所有协作都要由人自己去转述。你可以让一个 Agent 直接联系另一个 Agent，带着明确目标和回执要求去推进协作。

常见场景：

1. 总控通知执行通道开始某个批次
2. 结构治理要求某通道补齐真源或字段
3. 前端通道联系运行时通道确认接口或状态
4. 测试通道向执行通道回传阻塞点

### 3.6 让协作形成闭环
Qoreon 支持带 `callback_to` 和 `task_with_receipt` 的消息模式。
这意味着：

1. 消息不是发出去就结束
2. 处理完可以回给原发送 Agent
3. 回执、run、产出路径可以被继续消费
4. 总控和主责人不必靠人工翻聊天记录追进度

## 4. 一件事情在 Qoreon 里通常怎么被推进

一条标准工作链路通常是这样：

1. 产品或用户提出目标
2. 总控确定主线、优先级和边界
3. 架构师或技术门禁角色判断技术路线与约束
4. 主责通道接任务，主负责 Agent 持续推进
5. 需要协同时，Agent 给其他 Agent 发结构化消息
6. 产出物逐步形成，关键证据被保留下来
7. 测试/验收/总控做最终收口

所以 Qoreon 不是“所有人围着一个聊天框转”，而是：
**每个人围绕同一条主线在不同层次上推进，并且能互相接力。**

## 5. 我们推荐怎么组织工作

### 5.1 总控层：控制主线，不包办执行
总控适合负责：

1. 当前主线和批次编排
2. 优先级判断
3. 是否放行、是否收口
4. 跨板块冲突仲裁

总控不应该长期替代别人写实现、补页面、补字段。

### 5.2 架构师 / 技术门禁层：控制边界，不替代全体研发
技术层适合负责：

1. 技术路线判断
2. 契约与边界冻结
3. 关键变更门禁
4. 发布、回滚、兼容性判断

这层的重点是“控制技术秩序”，不是替所有人持续写代码。

### 5.3 业务主责通道：承接长期板块推进
一个成熟项目，应该让每个核心板块都能说清楚：

1. 主负责通道是谁
2. 主负责 Agent 是谁
3. 备份 Agent 是谁
4. 需要哪些支撑通道

这样做的好处是：
- 知识沉淀连续
- 问题出来后先找板块主责
- 总控和架构不会被迫长期下沉救火

### 5.4 横向支撑通道：提供公共能力，不替代业务主责
例如：

1. 运行时 / API / CCB
2. 前端实现与可见结果
3. 数据契约与兼容治理
4. 测试、验收、放行
5. 结构治理、技能治理

横向支撑的职责，是把公共能力做稳，而不是长期替业务通道推进业务本身。

### 5.5 PMO / 日常管理层：管节奏、催闭环、清积压
日常管理适合负责：

1. 任务节奏管理
2. 催办与回收
3. 问题聚合
4. 待验收积压和状态清理

它解决的是“系统如何不失速”，不是替代总控做战略判断。

## 6. 你应该怎么理解“跨项目协作”

Qoreon 不是只能在单个项目内部转。
它可以把项目之间的协作也组织起来，但前提是：
**跨项目协作必须有结构化桥接，不允许混写真源。**

### 6.1 为什么需要跨项目协作
常见场景包括：

1. 生产主项目和开发调试项目之间的联动
2. 主系统和开源执行项目之间的桥接
3. 一个业务项目向另一个业务项目请求能力支撑
4. 内部项目与外部执行项目之间的协作

### 6.2 跨项目协作怎么成立
推荐方式不是“大家直接混着聊”，而是：

1. 明确来源项目和目标项目
2. 明确来源通道、目标通道、主负责 Agent
3. 带 `callback_to`，确保结果能回给发起方
4. 用桥接通道或桥接角色处理项目间链路

这样，跨项目协作仍然可回溯、可验证、可收口。

### 6.3 跨项目协作不该做什么
不应该：

1. 把 A 项目的真源直接写到 B 项目里
2. 把不同项目的运行时上下文混在一起
3. 在没有桥接规则时直接复用内部敏感字段
4. 让跨项目消息失去来源和回执目标

## 7. 推荐的三种工作方式

### 7.1 单项目主线模式
适合大多数日常功能推进。

做法：
1. 总控定主线
2. 架构定边界
3. 主责通道推进
4. 支撑通道协作
5. 测试与验收收口

### 7.2 多 Agent 联动模式
适合一件事需要多个通道配合时。

做法：
1. 主负责 Agent 持主线
2. 对外发结构化消息给相关 Agent
3. 要求回执与证据路径
4. 回到主负责 Agent 汇总判断

### 7.3 跨项目桥接模式
适合生产、开发、开源执行等多项目并行时。

做法：
1. 项目之间用桥接角色或桥接通道沟通
2. 保留来源项目与目标项目身份
3. 让回执回到正确项目的正确 Agent
4. 防止真源和上下文混写

## 8. 最后一句总结

如果你从整体上理解 Qoreon，它不是“会聊天的任务系统”，也不是“会列任务的消息系统”。
它真正要做的是：

**让项目、通道、Agent、任务、消息、验收和跨项目协作都进入同一套可继续推进的工作机制。**

看懂这一点，你再进入某个具体项目，就不会只看到一堆目录和对话，而会知道：
- 这套系统里谁在负责什么
- 现在可以做什么动作
- 一件事会怎样流转
- 为什么不同角色要按不同层级工作
