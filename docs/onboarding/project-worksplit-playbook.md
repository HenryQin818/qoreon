# 基础项目开发如何做分工：Qoreon 通道分工方法

适用对象：
- 正在搭建一个新项目，但不知道该怎么分工的人
- 已经有很多 Agent，但协作越来越乱、总控和架构开始下沉包办的人
- 想把 Qoreon 当前的通道体系看懂，并抽出一套可复用方法的人

## 1. 先说结论：分工不是越多越好，而是越稳定越好

一个基础项目开发，最怕两件事：

1. 没有分工，所有事情都回到总控和架构。
2. 分工过度，问题刚出现就新增一个通道，最后谁也说不清谁对什么负责。

所以这篇文章不讨论“怎么把通道堆满”，而是讨论：

**一个项目最少应该怎么分工，什么时候该扩，扩了之后每一层应该承担什么责任。**

## 2. 推荐采用：2 + 4 + X 的分工模型

### 2.1 两个中央角色

每个项目至少需要两个中央角色：

1. 总控 / 产品主线
2. 架构 / 技术门禁

这两个角色负责：
- 控目标与优先级
- 控技术边界与门禁
- 控冲突裁决与最终收口

这两个角色**不应该长期替代执行通道做实现**。

### 2.2 四个核心执行通道

一个基础项目，通常至少要有这四类核心执行通道：

1. 运行时 / 后端
2. 前端 / 交互
3. 数据 / 契约
4. 测试 / 验收

这四类通道已经覆盖了绝大多数软件项目的基本工作面。

### 2.3 X 个按需扩展通道

当项目复杂度上升时，再扩这些按需通道：

1. PMO / 排期督办
2. 结构治理
3. 技能治理
4. 原型 / 需求承接
5. 项目运维
6. Git / 开源管理
7. 跨项目桥接
8. 自动化巡检
9. 可视化看板
10. 用户镜像 / 业务判断支撑

这些通道的意义，是把公共能力和复杂协作从主线执行里剥出来，而不是再造一套主线。

## 3. 每一类通道分别负责什么

### 3.1 总控 / 产品主线

负责：
1. 决定现在要推进什么。
2. 决定优先级与批次顺序。
3. 决定是否放行、是否收口、是否升级。

不负责：
1. 不长期替代前端、后端、数据、测试做具体实现。
2. 不把所有问题都留在自己这里集中处理。

输入：需求、风险、验收结果、协同回执。
输出：主线任务、推进指令、放行结论。

### 3.2 架构 / 技术门禁

负责：
1. 决定技术路线。
2. 决定结构边界和契约约束。
3. 审核高风险变更、回滚、兼容与发布门禁。

不负责：
1. 不替每个执行通道持续写代码。
2. 不承担所有模块的长期日常推进。

输入：方案、实现差异、风险项、关键设计争议。
输出：技术口径、门禁规则、边界裁决。

### 3.3 运行时 / 后端通道

负责：
1. 服务端逻辑、调度、运行链路。
2. API、并发、权限、安全、执行环境。
3. 与外部运行时或执行引擎的接入。

不负责：
1. 不定义产品主线。
2. 不决定页面信息架构。
3. 不替测试做最终放行。

输入：需求、接口约束、架构门禁。
输出：后端实现、接口能力、运行证据。

### 3.4 前端 / 交互通道

负责：
1. 页面结构、交互呈现、用户操作路径。
2. 可见结果、导航组织、信息分层。
3. 页面级问题修复与体验收敛。

不负责：
1. 不决定服务端策略。
2. 不私自扩改契约。
3. 不替总控决定业务优先级。

输入：业务目标、原型、接口约定。
输出：页面、交互、可见验证结果。

### 3.5 数据 / 契约通道

负责：
1. 数据结构、字段口径、契约校验。
2. 兼容策略、白名单、回滚约束。
3. 多模块之间的数据一致性治理。

不负责：
1. 不替前后端推进全部实现。
2. 不替总控判断业务是否通过。

输入：接口变更、字段需求、历史兼容问题。
输出：契约规则、字段规范、校验结论。

### 3.6 测试 / 验收通道

负责：
1. 功能验证、回归、独立终验。
2. 最小放行检查。
3. 给出是否通过、唯一阻塞和发布意见。

不负责：
1. 不替执行通道补实现。
2. 不把“实施方自测通过”直接当最终结论。

输入：交付物、页面、接口、运行环境。
输出：验收结论、阻塞、放行意见。

### 3.7 PMO / 排期督办

负责：
1. 跟节奏。
2. 催回执。
3. 清积压。
4. 做跨批次推进协调。

不负责：
1. 不替总控做战略判断。
2. 不替执行通道做技术决策。

### 3.8 结构治理 / 技能治理 / 原型承接 / 运维 / Git / 桥接

这些都属于横向支撑通道。
它们负责“让系统更稳定、更好协作、更可复制”，不负责替代主线执行。

可以理解为：
- 结构治理：管目录、真源、契约挂载
- 技能治理：管模板、技能、协作规范
- 原型承接：管需求收集、方案整理、原型表达
- 项目运维：管运行状态、异常修复、会话健康
- Git / 开源管理：管仓库、同步、开源收口
- 跨项目桥接：管项目之间的消息链路和回执回收

## 4. 什么时候应该新建通道，什么时候不应该

这是最关键的一章。

### 4.1 应该新建通道的情况

满足下面任意两条，才建议新建通道：

1. 这类工作会长期存在，不是一轮任务。
2. 它有稳定输入和稳定输出。
3. 它需要独立知识沉淀。
4. 它会反复和多个主线发生协作。
5. 继续塞在现有通道里会导致责任面混乱。

### 4.2 不该新建通道的情况

下面这些情况，通常不该新建通道：

1. 只是一个临时任务。
2. 只是某个通道里的一个专项动作。
3. 只是某个主负责 Agent 忙不过来。
4. 只是要补一轮讨论或验收。

这几种情况，更适合：
- 新建任务
- 新建专项 Agent
- 新建候选 Agent
- 发起一次跨通道协作

### 4.3 一个简单判断公式

如果它是“长期职责”，建通道。  
如果它是“同一职责下的不同执行者”，建 Agent。  
如果它是“这轮要做的一件事”，建任务。

## 5. Qoreon 当前体系，如何映射到这套分工模型

### 5.1 中央层

当前比较明确的中央层包括：

1. `主体-总控（合并与验收）`
2. `子级01-Build引擎（扫描-解析-聚合-渲染）`

其中：
- `主体-总控` 更偏主线、验收、收口。
- `子级01` 当前实际承接了较强的架构与系统组织判断。

### 5.2 核心执行层

这是 Qoreon 当前最像“基础项目开发四件套”的部分：

1. `子级02-CCB运行时`：运行时 / server / 并发 / 安全 / 启动
2. `子级04-前端体验`：页面 / 交互 / 展示结果
3. `子级06-数据治理与契约`：字段、契约、规格校验
4. `子级08-测试与验收`：功能、回归、发布验收

这四个通道基本已经组成了一个软件项目最小可运行执行面。

### 5.3 平台增强层

这类通道不是基础四件套，但对当前 Qoreon 很重要：

1. `子级03-多CLI适配器`
2. `子级05-任务巡检与留痕`
3. `子级07-可视化看板`

它们承担的是平台差异化能力，而不是基础软件交付本身。

### 5.4 支撑治理层

当前 Qoreon 的治理支撑比较完整：

1. `辅助01-项目结构治理`
2. `辅助02-团队协作Skills治理`
3. `辅助04-原型设计与Demo可视化`
4. `辅助05-督办PMO`
5. `辅助06-项目运维`
6. `辅助07-Git仓库管理`

这说明 Qoreon 已经不是一个“刚起步的小项目”，而是进入了需要治理配套的阶段。

### 5.5 特殊协作层

Qoreon 里还有两类很特别的通道：

1. `辅助03-用户镜像`
2. `辅助08-开源执行桥接`

它们不是传统研发项目里的必备通道，但在当前系统中非常有价值：
- 一个负责业务判断与规划支撑
- 一个负责跨项目桥接与执行链路对齐

## 6. 一条需求应该怎样穿过这些通道

推荐链路如下：

1. 用户或产品提出目标。
2. 总控确定主线、优先级和主负责通道。
3. 架构 / 技术门禁确定技术边界。
4. 核心执行通道开始推进。
5. 需要时调用支撑治理通道。
6. 测试 / 验收收口。
7. 若牵涉多项目，再由桥接通道建立跨项目协作链路。

这条链路的重点，不是“所有通道都要参与”，而是：

**谁是主线，谁做支撑，谁做门禁，谁做收口，必须分清楚。**

## 7. 这套分工里最常见的坏模式

最常见的 6 个坏模式：

1. 总控长期包办执行。
2. 架构师变成所有模块的长期实施者。
3. 支撑通道替代业务主责推进业务。
4. 一个问题出现就新增通道。
5. 跨项目直接混写真源。
6. 过程协作全靠反馈文件，消息链路反而断掉。

## 8. 一句话建议

如果你在做一个基础项目开发，推荐先从这套结构起步：

1. `总控 + 架构` 两个中央角色
2. `后端 + 前端 + 数据 + 测试` 四个核心执行通道
3. 再按项目复杂度增加 `PMO / 运维 / 原型 / 技能 / 桥接 / Git / 巡检` 等横向支撑

先把责任面稳定下来，再追求通道数量和组织复杂度。

## 9. 一句话总结

**分工不是为了看起来专业，而是为了让项目在复杂之后仍然能继续推进。**

真正有效的分工，不是通道越多越高级，而是：
- 主线有人负责
- 支撑有人承接
- 门禁有人把关
- 收口有人拍板
- 跨项目时也不会把链路搞乱
