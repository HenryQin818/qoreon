# 一个公开候选包，如何穿过三个项目：Qoreon 跨项目协同现场

这不是一个“系统能力介绍”，而是一段最近真实发生过的协作。

故事里有三个地方：
- 内部主工程：每天还在继续开发和修问题的地方。
- 开源执行项目：专门把能公开的内容整理出来的工作台。
- 公开仓库：最后交给外部人使用和查看的版本。

这次协作的目标很具体：
把内部已经跑起来的一部分能力，整理成一个真正能给外部试用的公开候选包。

## 第一幕：先把入口收成一个

最早暴露出来的问题，不是代码，也不是文档，而是大家一度不知道该在哪个项目里继续干活。

和开源有关的事情，原先分散在台前工作台和后台筹备区两个地方。一个地方负责日常可见操作，一个地方保留了很多筹备材料和迁移痕迹。短期看起来都能用，但人一多，就会出现最现实的问题：任务到底算在哪儿接，结果到底回到哪里。

团队先做的不是导仓，也不是补 README，而是先把入口收成一个。台前只留一个真正承担日常推进的开源执行项目；后台那边退回去，只保留宿主、留痕和迁移支持职责，不再继续承担日常派发。

这是这次协作真正的起点。因为从那一刻开始，大家终于不用再反复确认“这件事到底该在哪儿说”。

### 这一幕，系统里用了什么
- 多项目并行：内部主工程、开源执行项目、公开仓各自有边界。
- 项目内分工：总控、结构治理、桥接位各自处理不同判断。
- 结构化消息：不是只发一句“你去处理一下”，而是带着目标、回执和下一步要求去协作。

## 第二幕：不能把正在施工的现场直接往外发

入口收清楚之后，第二个问题也很快摆上桌面：内部主工程还在继续变化。

有人在修页面，有人在补脚本，有人还在收缩结构。如果这时候直接把当前目录拿出去，外部看到的就不是一个整理过的版本，而是一片还在施工的现场。于是负责仓库同步的那位 Agent 很快把规则卡死：不能直接拿当前工作树外发，必须先冻结出一轮可追溯的公开快照。

这轮快照不是一种感觉，而是一份明确的对象。它知道自己包含什么、不包含什么，也知道后面的导出、脱敏、文档补充和测试验收都应该围着谁来做。

从这个时候开始，团队讨论的就不再是“哪个目录比较新”，而是“这一轮冻结出来的东西，够不够成为公开候选”。

### 这一幕，系统里用了什么
- 跨项目桥接：把内部主工程里的进展有边界地交给开源执行项目，而不是混写真源。
- 任务与消息配合：任务承接阶段目标，消息推动具体动作。
- 证据留痕：冻结批次、导出结论、回执信息都能回看。

## 第三幕：几条线同时开工，但没有人需要盯住全部细节

等公开快照出来以后，开源执行项目里很快分成了几条并行线。

一条线盯边界，负责判断哪些内容能带出去，哪些内容只能留在内部。一条线盯文档和示例，把内部人能看懂的材料改写成外部人也能照着走的说明。还有一条线盯测试和最小启动链，确保这不是一个只会展示页面的壳，而是真的能让别人试起来。

这时候多 Agent 协作的价值就出来了。它不是所有人围着同一个聊天窗口打转，而是每个人只盯自己该盯的那一小段：谁负责导出，谁负责说明，谁负责验收，都非常清楚。真正需要补信息时，也不是等人来回传话，而是 Agent 之间直接发消息、带回执要求去推进。

所以这一步最重要的，不是“大家都很忙”，而是忙得不乱。每一条线都知道自己的输入是什么，输出要交给谁。

### 这一幕，系统里用了什么
- 一个项目里多个通道并行：导出、文档、测试可以同步推进。
- 给 Agent 安排具体任务：不是泛化讨论，而是分配明确动作。
- Agent 互发消息：缺信息时直接协作，不靠人当中转站。
- 产出物做证据：文件、结论、路径都能被后续继续消费。

## 第四幕：真正的检验不是内部觉得差不多，而是陌生人能不能走完一遍

做到这里，最容易出现一种错觉：页面已经有了，说明也补了，示例项目看上去也能用了，好像这件事已经差不多了。

但团队很清楚，内部人看着顺手，不等于外部人第一次拿到时也能走通。所以测试与验收那条线在这里变得非常关键。它要看的不是“像不像完成了”，而是最真实的几个问题：初始化路径能不能跑通，最小示例能不能用，一条基本协作链能不能走完，失败时别人能不能知道卡在哪里。

这一步把整个故事从“内部已经理解的成果”，推进成“外部也能接住的产品”。

### 这一幕，系统里用了什么
- 测试 / 验收 Agent：专门负责判断能不能放行。
- 运行记录与回看：每次尝试做了什么，都能被回顾。
- 结构化回执：结果会回到发起这轮协作的人那里，而不是散落在多个对话里。

## 第五幕：协作成立，不是因为都做了事，而是因为结果收回到了一个地方

最后真正让这轮协作成立的，不是每条线都交了一点东西，而是这些结果重新回到了一个统一的地方。

有人交回导出和脱敏结论，有人交回文档和示例状态，有人交回测试结果，最后由开源执行项目里的总控来判断：这一轮公开候选到底能不能成立，还差哪一块，下一步应该谁继续动。

这也是这次故事最想说明的一点：多项目协作不是“多开几个项目，多放几个 Agent”就自动成立。它成立的前提是，入口清楚，边界清楚，结果回得来。

Qoreon 在这个场景里提供的，不只是让多个 Agent 同时工作，而是让这些工作最后能重新被组织起来，变成一个外部人真的能看懂、真的能用起来的公开版本。

## 这个故事里，观众最该看懂的系统能力

1. 一个平台里可以同时有多个项目，并且每个项目职责不同。
2. 每个项目里可以放多个 Agent，而且它们可以按通道分工协作。
3. Agent 可以直接互发消息推进工作，不必全都经过人工转述。
4. 任务、消息、产出物各自承担不同角色：主线、过程、证据。
5. 跨项目协作不是复制粘贴，而是通过桥接和回执把结果有边界地接力下去。
